# VLM 기반 지능형 매크로 엔진 개발 계획

과거의 단순 좌표 기반 매크로(Record & Replay) 방식의 한계를 극복하고, 소형 VLM(Vision-Language Model, 예: Qwen 3.5 0.8B)을 활용하여 화면의 문맥을 이해하고 동적으로 액션을 수행하는 **'지능형 VLA(Vision-Language-Action) 매크로 엔진'**의 실현 가능성 검토 및 개발 계획입니다.

## 💡 실현 가능성 검증 (Feasibility Check)

최근 연구 동향(tavily 검색 기반)과 소형 모델의 발전에 따르면, 이 프로젝트는 **소비자용 GPU 환경에서 충분히 실현 가능**합니다.

1. **하드웨어 및 VRAM 최적화**: 
   - Qwen 0.8B ~ 3B 수준의 소형 VLM은 양자화(4-bit/8-bit) 적용 시 **1GB ~ 3GB 내외의 VRAM**만을 소모하므로, 최근 일반적인 소비자용 GPU(RTX 3060, 4060 등 8GB 이상) 환경에서 배경 프로세스로 무리 없이 실행할 수 있습니다.
2. **반응 속도(Latency)**:
   - vLLM, TensorRT-LLM 구동기나 llama.cpp를 활용할 경우 작은 VLM 모델은 토큰 생성 속도가 매우 빠릅니다. 화면(Vision) 인코딩에 걸리는 약 100~200ms를 제외하면 실시간에 가까운 반응(1초 내외 행동 결정)이 가능하여 단순 게임 숙제나 반복 퀘스트 수행에 적합합니다.
3. **GUI Grounding (시각적 좌표 추론)**:
   - 최근 Qwen-GUI-3B, Qwen2-VL과 같은 모델들은 화면 UI의 "의미(문맥)"를 이해하고 지정된 버튼의 "Bounding Box 좌표(`[x, y]`)"를 반환하는 능력이 탁월합니다. 단순히 "일일 퀘스트 버튼을 눌러"라고 명령하면 팝업 위치가 바뀌어도 올바르게 찾아 클릭할 수 있습니다.

## 🏗️ 아키텍처 및 핵심 워크플로우

엔진은 크게 **기록(Record)**, **학습 및 의미 추출(Learn)**, **실행(Play/Agent)**의 세 단계로 나뉩니다.

### 1단계: Recording (행동 기록)
- **개요**: 유저가 직접 게임을 플레이하며 숙제를 수행하는 과정을 백그라운드에서 추적합니다.
- **방식**: 
  - `pynput` 등의 라이브러리로 모든 키보드/마우스 입력 이벤트를 후킹합니다.
  - 마우스 클릭이나 트리거 키가 입력되는 시점의 **직전/직후 화면(Screenshot)**을 캡처하여 로컬 DB에 저장합니다.
  - 좌표 데이터(X, Y)와 입력된 키 정보를 이미지 데이터와 함께 매핑하여 로그(`trace.json`)로 남깁니다.

### 2단계: Semantic Extraction (문맥 추론 및 그래프화)
- **개요**: 저장된 기록(스크린샷 + 클릭 X, Y)을 VLM이 백그라운드에서 분석하여 단순 좌표가 아닌 '의미'로 변환합니다.
- **VLM 역할**: 
  - "이전 화면에서 클릭된 (X, Y) 지점에 어떤 버튼이 있었나?" -> `VLM: "일일 퀘스트 받기 버튼"`
  - 입력된 로그들을 묶어 **상태 전이 그래프(State Machine)**나 행동 스크립트(예: 메인화면 -> 퀘스트 수락 -> 사냥 -> 완료)로 추상화합니다.

### 3단계: Playback & Agent Execution (실행 및 자율 행동)
- **개요**: 만들어진 그래프나 스크립트를 기반으로 매크로를 실행하는 단계입니다.
- **방식**:
  - `mss` 등을 이용해 화면을 1~2 FPS로 계속 캡처하며 VLM에게 전달합니다.
  - VLM이 현재 화면의 상태를 파악 ("지금은 메인화면에 있음")하고 다음 목표 행동("일일 퀘스트 버튼 찾기")을 인식합니다.
  - VLM에게 **"현재 화면에서 '일일 퀘스트 버튼'의 위치 좌푯값을 반환해"** 라고 프롬프트를 전송하면, 모델이 `[X, 기Y]` 좌표를 반환합니다.
  - 파이썬(AutoGUI)이 해당 좌표를 실제 마우스로 클릭합니다. 화면이 로딩되어 위치가 조금씩 바뀌어도 문맥 기반으로 동작하므로 전혀 막힘이 없습니다.

## 🛠️ 권장 기술 스택 (Tech Stack)

* **언어 & 프레임워크**: Python (접근성과 라이브러리 생태계, AI 연결이 용이함)
* **VLM 모델**: Qwen2-VL (2B 사이즈) 또는 Qwen-GUI-3B (가장 추천하는 사이즈. GPU 메모리를 적게 먹고 GUI 인식률 최상위권)
* **LLM 백엔드**: `llama.cpp` 엔진 또는 `vLLM` (로컬 GPU 가속용)
* **화면 캡쳐**: `mss` (가장 빠르고 부하가 적은 스크린 촬영)
* **입력 감지 및 매크로 액션**: `pynput` (로우레벨 후킹), `pyautogui` / `pydirectinput` (DirectX 기반 게임 조작용)

## 🚀 개발 마일스톤 (Milestones)

### [Phase 1] MVP 스켈레톤 구축: Data Logger 구현
- [ ] 파이썬 기반 기초 키보드/마우스 입력 후킹 시스템 구축
- [ ] 클릭 시점 스크린샷 캡쳐 기능 및 `JSON` 포맷 기록 (X,Y 좌표 저장) 시스템 구현

### [Phase 2] 로컬 VLM 연동 및 추론 시스템 구축
- [ ] 로컬 환경에 Qwen2-VL 또는 소형 0.8B~3B 모델 구동 테스트 (HuggingFace / vLLM 연동)
- [ ] 입력된 스크린샷과 클릭 좌표를 VLM에게 줘서 의미를 해석("유저가 무슨 버튼을 눌렀는지") 하는 프롬프트 엔지니어링

### [Phase 3] Playback 엔진 구축
- [ ] 녹화된 '의미 정보(텍스트)'를 바탕으로 현재 화면에서 다시 좌표를 찾아 클릭하는 역추론 모듈 구현
- [ ] 실시간 스크린샷을 읽어오며 에이전트 루프가 동작하는지 테스트용 웹페이지 및 단순 UI 앱으로 검증

### [Phase 4] 게임 맞춤형 최적화 & 로버스트니즈
- [ ] 게임 클라이언트의 안티 치트를 우회하는 수준의 자연스러운 마우스 움직임(Curves) 추가 고려
- [ ] 로딩, 렉 등으로 화면 전환이 늦어질 경우 VLM이 판단하여 "대기(Wait)"를 수행하도록 상태 전이 고도화
- [ ] 소비자 편의를 위한 단순한 인터페이스(Tkinter 또는 CustomTkinter) 제작

> [!TIP]
> **성능 최적화 팁**: 매 프레임마다 VLM 추론을 돌리면 GPU 부하가 크고 게임 프레임 드랍이 발생할 수도 있습니다. 따라서 액션을 취한 뒤 일정 시간(예: 1~2초) 휴식하며 "화면 전환이 완료되었는지 차이를 비교(Pixel diff)" 한 후 차이가 클 때만 VLM을 호출하도록 설계하는 방안이 매우 효과적입니다.
